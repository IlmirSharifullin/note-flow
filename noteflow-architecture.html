Раздел 1 — ГОСТ 34 / ГОСТ 19

Техническое задание

1.1 Назначение системы

NoteFlow — веб-приложение для ведения личного дневника и создания структурированных заметок в формате Markdown. Целевая аудитория — индивидуальные пользователи, которым нужен личный цифровой блокнот: текстовые записи, вложенные файлы и изображения, организация по тегам. Система построена на трёх независимых микросервисах с хранилищем MinIO для файлов.

1.2 Цели разработки

  • Создать масштабируемую микросервисную платформу для не менее 10 000 активных пользователей в сутки
  • Обеспечить время отклика API p95 < 200 мс для операций чтения заметок
  • Реализовать асинхронное взаимодействие сервисов через событийную шину Kafka
  • Обеспечить хранение файлов объёмом до 100 МБ через MinIO S3-совместимое API
  • Предоставить полнотекстовый поиск по заметкам пользователя

Раздел 3

C4-диаграммы + Sequence Diagrams

C4 Level 1 — System Context

Пользователь взаимодействует с NoteFlow через браузер или мобильное приложение. Два внешних компонента: Email-провайдер (регистрация, уведомления) и MinIO/S3 (файлы и изображения к заметкам).

C4 Level 2 — Container Diagram

Три микросервиса на Python + FastAPI, каждый со своей БД. API Gateway — единая точка входа. Kafka — событийная шина. Redis — кэш, сессии, rate limiting.

C4 Level 3 — Note Service Components

Внутренняя структура Note Service: FastAPI-роутеры → NoteService (бизнес-логика) → NoteRepository (Motor/MongoDB) + EventPublisher (aiokafka). Redis — кэш заметок.

Sequence Diagram — Создание заметки

Синхронный HTTP-поток: клиент → Gateway (JWT check) → Note Service → MongoDB. Асинхронно: событие note.created в Kafka → File Service при необходимости обновляет ссылки в заметке.

Выбор брокера: Kafka vs NATS vs RabbitMQ

5.1 Сравнение

OrderingPer-queuePer-subjectPer-partition
Exactly-onceНетНетДа (EOS + idempotent)

5.2 Почему выбрали Kafka

Главный аргумент: событие user.registered одновременно потребляют Note Service (создать приветственную заметку) и потенциальный Notification Service (welcome-письмо). Kafka делает это нативно — каждая consumer group читает свой offset независимо. RabbitMQ потребует fanout-топологии, которая сложнее в поддержке.
  • Replay: при добавлении нового сервиса можно переиграть исторические события (например, пересоздать поисковый индекс)
  • Ordering: изменения одной заметки → одна партиция (ключ noteId) → гарантированный порядок при обработке истории
  • Outbox pattern: запись в MongoDB + outbox в одной транзакции; Debezium → Kafka
Честная оценка: для NoteFlow MVP с 10k DAU NATS JetStream объективно подошёл бы лучше — один бинарь без KRaft-кластера, sub-миллисекундная латентность, встроенный KV-store (замена части Redis). Закрывает ≈90% требований при 1/3 операционной сложности. Kafka оправдана при >100k RPS и зрелой аналитической экосистеме.

5.4 EDA-паттерны

Choreography Saga

Регистрация: User Svc → user.registered → Note Service создаёт приветственную заметку → File Service инициализирует квоту. Без центрального оркестратора.

Outbox Pattern

Запись в MongoDB и в коллекцию outbox в одной транзакции. Debezium читает outbox и публикует в Kafka. Гарантия: нет события без записи в БД.

Event-carried State Transfer

Событие file.uploaded несёт {fileId, noteId, userId, presignedUrl}. Note Service не делает HTTP-запрос в File Service — данные уже в событии.

Dead Letter Queue

Consumer: 3 retry с exponential backoff → топик *.dlq. Алерт в Grafana + ручная проверка через Kafka UI.

7.2 Стратегии

Cache-Aside

Для заметок и профилей: Redis miss → запрос к БД → записать в Redis. При изменении — явная инвалидация ключа.

Write-Through

Для user:{id}: PATCH профиля → обновляем PostgreSQL и Redis в одной операции. Консистентность гарантирована.

Token Bucket (Rate Limit)

Lua-скрипт: атомарный INCR+CHECK в одном roundtrip. При K8s — общий счётчик для всех инстансов Gateway.

Pub/Sub инвалидация

PATCH заметки → Note Service публикует в Redis channel invalidate:note → все инстансы сбрасывают локальный кэш.

7.3 Топология

Redis Sentinel (1 master + 2 replicas) — auto-failover для production без sharding. При нагрузке >100k ops/s — Redis Cluster (3+ шарда). Valkey 7.x поддерживает оба режима.
